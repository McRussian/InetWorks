# HTTP Request Builder
В данной задаче вам предлагается познакомиться с шаблоном проектирования под названием Builder Pattern через написание своего собственного "строителя" HTTP запросов. Интерфейс, похожий на тот, который вы будете реализовывать, встречается во многих современных языках программирования, например [Java](https://docs.oracle.com/en/java/javase/22/docs/api/java.net.http/java/net/http/HttpRequest.Builder.html).

HTTP - это протокол взаимодействия, который определяет структуру передачи данных между клиентом и сервером в интернете. Почти весь трафик на вашем компьютере так или иначе использует HTTP: браузер скачивает страничку того, что вы читаете, приложение для прослушивания музыки получает список треков, да даже операционная система может проверять доступные обновления через него.

## Что такое Builder Pattern?
Представьте объект, требующий сложной пошаговой инициализации множества своих полей, среди которых тоже могут оказаться тяжелые структуры. Тогда конструктор для соответствующего класса будет очень громоздким и малопонятным для стороннего разработчика. Самое просто решение - создать много подклассов для разных комбинаций параметров. Например, в данной задаче это были бы отдельные классы `GETRequest`, `POSTRequest`, `GETRequestWithHeaders` и так далее. Нетрудно заметить, что подобный подход слишком сильно увеличивает объем кода.

Builder Pattern предлагает делегировать конструирование нашего сложного объекта сторонним классам-строителям. Процесс создания разбивается на отдельные смысловые шаги, например `установить host`, `установить заголовки`, `задать параметры запроса`. Тогда, чтобы сконструировать объект, нужно поочередно вызывать методы "строителя". Причем совершенно необязательно вызывать все описанные методы, если используемая конфигурация этого не подразумевает.

Применяя данный шаблон проектирования в рамках нашей задачи, мы можем добиться подобного конструирования контента HTTP запросов:
```
HttpRequest req = HttpRequest::Builder()
        .SetPost("/endpoint")
        .SetHost("example.com")
        .SetQuery("q", "hello world")
        .SetHeader("User-Agent", "ReqBuilder/1.0")
        .SetHeader("Content-Type", "application/json")
        .SetBody("Test123")
        .Build();
```

Также этот паттерн подразумевает создание нескольких классов-строителей, каждый из которых реализует методы построения по-своему. В данной задаче эта и некоторые другие особенности Builder Pattern затронуты не будут. [Здесь](https://refactoringu.ru/ru/design-patterns/builder.html) можно прочитать об этом подробнее.

## Формат HTTP-запроса (HTTP/1.0–1.1)
В рамках данной задачи вам понадобится примерное представление о структуре запросов в HTTP/1.x. В основном используется текстовое представление со следующей структурой:
1. **Строка запроса**: `<METHOD> <request-target> HTTP/<major>.<minor>` и `\r\n` в конце.
2. **Заголовки**: по одному на строку в формате `<Name>: <Value>\r\n`. Обычно порядок и регистр не важен.
3. **Пустая строка**: `\r\n` отделяет заголовки от тела.
4. **Тело** (опционально): для запросов с телом обязательно выставляйте `Content-Length` с числом октетов.
Для простоты будем считать, что наш builder будет создавать только запросы HTTP версии 1.1.

Подробнее и с примерами о структуре запросов можно почитать на [MDN](https://developer.mozilla.org/en-US/docs/Web/HTTP/Messages).

## Percent-encoding и RFC 3986
При формировании запроса также можно указать query-параметры для URL. Это набор пар "ключ-значение", которые добавляются вместе со знаком `?` после адреса страницы. Такие дополнительные параметры позволяют передать на сервер дополнительную информацию, не задействуя тело запроса.

Пример: ```https://api.example.com/articles?page=3&per_page=20```.

Как нетрудно заметить, данный формат передачи данных может вызвать проблемы, если в одном из ключей встретится неудобный символ, например пробел или `?`. В реальной жизни эта проблема решается с помощью [percent-encoding](https://developer.mozilla.org/en-US/docs/Glossary/percent-encoding). Вам достаточно знать следующие простые правила кодирования символов:

* Незарезервированные символы `[A-Za-z0-9-._~]` остаются как есть.
* Остальные кодируются как `%HH`, где `HH` — две заглавные шестнадцатеричные цифры ASCII-кода байта.

## Что нужно реализовать?
* `SetMethod()` - задает используемый метод для отправки запроса. Должны поддерживаться `GET`, `HEAD`, `POST`, `PUT`, `DELETE`, `PATCH`, `OPTIONS`. В данной задаче между ними нет отличий, кроме названия.
* `SetTarget()` - задает путь внутри host'а. Нормализуйте значение: пустая строка превращается в `/`, а относительный путь дополняется ведущим `/`.
* Методы `Get()`, `Head()` и так далее служат красивой оберткой над `SetMethod()` и `SetTarget()`.
* `SetHost()` и `SetPort()` - формируют значение заголовка `Host`. При заданном порте используйте формат `host:port`. В данной задаче в аргументе метода `Host()` не может содержаться адрес с указанным портом.
* `SetHeader()` - добавляет/заменяет заголовок без учета регистра. `RemoveHeader()` - удаляет его.
* `SetBody()` - устанавливает тело запроса и синхронизирует заголовок `Content-Length`. `SetNoBody()` - очищает тело и убирает `Content-Length`. Важно понимать, что изначально тело просто пустое, но `Content-Length` просто равен `0`.
* `SetQuery(key, value)` - добавляет percent-encoded параметр. `RemoveQuery(key)` - удаляет параметр по ключу. `ClearQuery()` - очищает параметры, добавленные билдером, но сохраняет путь из `Target()`.
* `Reset()` - возвращает билдер к значениям по умолчанию.
* `Build()` - создает `HttpRequest` с доступом к данным через геттеры и `ToString()`. Метод `ToString()` должен возвращать строку, полностью соответствующую описанному формату HTTP запроса.
* В классе `HttpRequest` для получения списка заголовков используется `HttpHeaderCollection`. Это вспомогательная [case intensive](https://en.wikipedia.org/wiki/Case_sensitivity) структура, логику которой вам придется реализовывать самостоятельно. В данной задаче вам предложено несколько методов, которые будут служить основным интерфейсом этого класса: `GetValue()`, `Set()`, `Remove()`, `Size()`, `Contains()`, `Items()`.

Обратите внимание, что в предложенном интерфейсе класса `Builder` все конструирующие функции возвращают `Builder&`. Именно это позволяет делать каскадные вызовы методов, как в примере.

### Возможно, неочевидные моменты

* Заголовок `Host` может быть задан как через `SetHost()`, так и через общий `SetHeader("Host", ...)`, он также может быть удален, как и обычный заголовок. Пустое значение допустимо, в этом случае строка запроса должна содержать `Host: ` без имени, а признак порта сбрасывается.
* В структуре `HttpHeaderCollection` сохраняйте последний использованный регистр, использованный для изменения.
* `SetTarget()` всегда приводит путь к виду с ведущим `/`, даже если пользователь передал относительное значение или пустую строку. Тесты проверяют оба сценария.
* `RemoveQuery()` не должен оставлять разрывов в итоговой строке запроса и обязан работать независимо от порядка добавления параметров.
* Для хранения информации о порте вам предлагается воспользоваться [std::optional](https://en.cppreference.com/w/cpp/utility/optional.html). Это контейнер, который может либо содержать значение, либо не содержать ничего. Изучить его интерфейс вам нужно самостоятельно.

Если у вас возникают вопросы по поведению программы в необычных ситуациях, то посмотрите на тесты. Вероятнее всего, они там описаны.
